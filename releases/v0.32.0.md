# v0.32.0 — 2026-06-17

The wave engine splits into a composable `wave-build` workflow driven by a thin group scheduler; a two-gate test discipline collapses redundant full-suite runs; per-pipeline isolated test infra lets pipelines QA in parallel; a global failure-biased test-output filter keeps healthy output out of agent context; plus `/jc:wave`, `/dev restart`, and the `p:tokens` token-attribution skill.

## Added

- **Mechanics: `workflows/wave-build.js` — composable per-pipeline build workflow.** The build mechanics for ONE pipeline (plan → architect → develop → QA → code review → GATE-1 → merge → GATE-2 → docs) now live in their own saved workflow, launched via `Workflow({name:'wave-build'})`. It is nesting-safe by construction: it makes zero `workflow()` calls, so the scheduler can compose it. `meta.phases` declares every build stage so `/workflows` renders the full look-ahead under a nested group.
- **Tier A: `/jc:wave` (`commands/jc/wave.md`) — batch hotfix/feature execution on `main`.** Resolve + pre-flight like `/wave`, then a grouping rule INVERTED from `/wave` (no worktree isolation → group for filesystem safety: parallelize only file-and-resource-disjoint tasks, serialize the rest), parallel implementation agents, an end-of-wave QA pass (POST-MERGE mode, full suite, zero-tolerance), one documenter, and a gitter commit. Ships as a flat `commands/jc.md` + a `commands/jc/wave.md` subcommand (no `jc/SKILL.md`).
- **Mechanics: `/dev restart [service]` — bounce one service.** `dev.sh` gains roster-driven `kill_server` / `start_server` / `cmd_restart_service` helpers as the single source of per-service launch/kill, removing the prior triplication of the launch strings; `commands/dev.md` documents the subcommand.
- **Mechanics: `p:tokens` skill (`commands/p/tokens/`).** Per-agent, per-operation, and per-workflow-run token attribution mined from local Claude Code JSONL transcripts — the runtime spend view OpenTelemetry can't give. Ships all three files: `SKILL.md`, the zero-dependency `token-ledger.mjs` tool, and `README.md`. (The tool is real and runnable, not a doc-only stub.)
- **Scripts: global `PostToolUse(Bash)` test-output filter (`settings.json` → `filter-test-output.sh`).** Wired globally so every agent's test output is filtered failure-biased — passing noise is suppressed, failures + summaries surface. Healthy output no longer enters agent context.

## Changed

- **Mechanics: wave engine SPLIT — scheduler vs. build.** `workflows/wave-pipelines.js` is reduced to a thin group scheduler: it sequences groups, runs pipelines within a group in parallel, handles `dependsOn` deferral and the durable STATE scribe, and calls `workflow('wave-build')` once per pipeline. All per-pipeline build mechanics moved to `wave-build.js`. One-level composition, nesting-safe.
- **Tier A: two-gate test discipline (`CLAUDE.md` + per-project `qa.md` / `developer.md`).** The full suite (unit + integration/e2e) now runs at EXACTLY two zero-tolerance gates per pipeline — GATE-1 (pre-merge full, after the fix loop + code review converge) and GATE-2 (post-merge full) — while developer self-QA, every fix-loop round, and code-review fixes run TARGETED (unit + typecheck + lint + only the failing/affected/adversarial profile). Collapses the prior ~5–8 full-suite runs per pipeline to 2 with no loss of shipped-code rigor. Agents are scope-aware: `TARGETED | FULL | POST-MERGE`.
- **Mechanics: per-pipeline ISOLATED test infra.** The build + QA agents bring up an isolated test stack per pipeline (isolated test DB / queue + allocated ports from the worktree's port file) instead of sharing one stack behind a lock, so pipelines in a wave QA in parallel without contention.
- **Tier A: `/jc` routing relaxed — no size ceiling.** Multi-part changes, dependencies, and refactors are allowed on `main`; worktree isolation + QA-before-merge become a deliberate choice (`/wave:build` / `/wave`), not something a change is forced into by size.
- **Tier A: `commands/build.md` + `docs-commands/.../build-reference.md`** re-framed as the declared-copy of the `wave-build` workflow, with the GATE-1 / GATE-2 step map and the per-pipeline-infra setup.

## Fixed

- **Mechanics: `wave-pipelines.js` — coerce a JSON-string `args` before validation.** The harness can deliver `Workflow({name, args})` payloads string-encoded, which made the engine throw a missing-`groups` error before any agent ran — a wave dying at the door after full pre-flight. The engine now parses-then-validates at the input boundary (`if (typeof args === 'string') args = JSON.parse(args)`).
- **Scripts: `filter-test-output.sh` — matcher broadened + wired globally.** The filter previously matched only bare runners (`pnpm test`, `pytest`) and was wired only in the QA wrappers, so the commands projects actually run (`pnpm run test:coverage`, `pnpm run integration`, `npm test`, playwright e2e) streamed unfiltered into context, and developer/fix-loop agents had no filter at all. Matcher now covers the real runners (pnpm/npm `run test|integration|e2e`, cucumber, playwright, jest/mocha, pytest) and the hook is global.

## Migration

#### → For adopters with an installed blueprint:

- **Wave engine split.** `/pcm update` adds `workflows/wave-build.js` and rewrites `workflows/wave-pipelines.js` into the thin scheduler. If you customized the old monolithic `wave-pipelines.js`, re-apply your changes against the split: per-pipeline build mechanics now live in `wave-build.js`, group scheduling in `wave-pipelines.js`. `commands/build.md` and `commands/wave.md` update to match (a declared copy of the workflow — keep them in sync with the JS).
- **Two-gate QA cadence (behavioral).** Your `qa.md` / `developer.md` and the `CLAUDE.md` Zero-Tolerance rule change: developer self-QA and fix-loop rounds run TARGETED; the full suite runs only at GATE-1 (pre-merge) and GATE-2 (post-merge). If you mandated "full suite every round," adopt the scope-aware split. Pair it with the global filter: add the `PostToolUse(Bash) → filter-test-output.sh` entry to your `settings.json`.
- **New commands / scripts.** `/jc:wave` (`commands/jc/wave.md`), `/dev restart` (`dev.sh` helpers + `commands/dev.md`), and the `p:tokens` skill (`commands/p/tokens/` — three files; `token-ledger.mjs` is the working tool). If your install registers commands in a contract/check script, add `jc/wave` and `p/tokens`.
